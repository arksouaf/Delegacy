Data Flow: Creating a Session

1. Developer app calls SDK вҶ’ createSession(account, policies, ttl)
2. SDK generates ephemeral keypair (session key)
3. SDK encodes registerSessionKey() call + policy configurations
4. SDK wraps everything in a UserOperation вҶ’ signed by MASTER KEY (one time)
5. UserOp вҶ’ Bundler вҶ’ EntryPoint вҶ’ Smart Account
6. Smart Account вҶ’ SessionKeyValidator.registerSessionKey()
7. Smart Account вҶ’ Each Policy.configure()
8. Session key is now active on-chain
9. SDK returns ActiveSession to the app (includes session private key)

Data Flow: Using a Session Key

 1. App calls SDK вҶ’ execute(session, target, data)
 2. SDK encodes the call into a UserOperation
 3. UserOp is signed by the SESSION KEY (not master key!)
 4. UserOp вҶ’ Bundler вҶ’ EntryPoint вҶ’ Smart Account
 5. Smart Account вҶ’ SessionKeyValidator.validateSessionKeyOp()
 6. Validator вҶ’ signature recovery вҶ’ checks time bounds
 7. Validator вҶ’ loops through all attached Policies
 8. Each Policy вҶ’ checkPolicy() вҶ’ returns allow/deny
 9. If all pass вҶ’ Smart Account.execute() вҶ’ target contract
10. After execution вҶ’ each Policy.updatePolicy() (spending tracking, etc.)

  в””в”Җв”Җ GasLimitPolicy       вҶ’ total gas budget

Security Model

Layer Protection
Smart Account Only EntryPoint or owner can execute
Session Validator ECDSA signature verification + time bounds
Policies On-chain enforced вҖ” no server trust required
EntryPoint Nonce prevents replay attacks
Paymaster Optional gas sponsorship (doesn't affect custody)
Master Key Never transmitted вҖ” signs only once to install session
